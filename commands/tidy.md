## 流程：扫 → 列方案待确认 → 执行

### 1. 扫 + 分类（递归）

按目录角色分析：
- **根目录**：散乱文件应归 `docs/` `data/` `deliverables/`
- **deliverables/**：只留最终交付版本
- **docs/**：清理解包残留 / 旧版本 / 重复
- **data/**：基本不动（仅清明显临时文件）

### 2. 识别 4 类问题

**A. 垃圾文件（直接删，不问）**
- Office 临时：`~$*` `.~lock.*`
- macOS 系统：`.DS_Store` `Thumbs.db`
- docx 解包残留：含 `word/` `_rels/` `[Content_Types].xml` 的目录
- hash 命名 png：`a354070ccd654ed08b27245268f0e84b.png`
- 转换副产品：同文档的 `.rtf` `.txt`（当 `.docx` + `.pdf` 都在时）

**B. 版本链（待确认）**

按时间排：日期大 = 新（`1126 < 0123 < 0313`）；`成品/final/定稿` = 最终；`_ztl/_dh/校核后` = 中间审核版；`-1/-2` 后缀 = 小修。最终版的 `.docx` + `.pdf` 都保留。最终版的 `.md` 若是 docx 导出工具产物（不是手写），可删。

**C. 散乱归位**

合同/审查意见/会议记录 → `docs/`；最终方案 → `deliverables/`；数据表 → `data/`；发票 → `docs/`。

**D. 命名归一（仅同级混风格时）**

触发信号：前缀有/无混用 / 分隔符混用（`-` vs ` ` vs 无）/ 脏后缀（`修改后` / `copy` / `2` / `-final`）/ 类型前缀不一 / 顶层散文件 + 同名子目录并存。

对每个 hot spot 给对照表：

```
## <目录>/ 命名归一
| 现名 | 问题 | 建议 |
|---|---|---|
| 浙水设计水效评估 | 无连字符 | 浙水设计-水效评估分析系统 |
```

软件名/正式名要用户确认（不机械推，可能漏字）；`git mv` 走 git 跟踪文件，普通 `mv` 走 `.gitignored`；模板/通用文件加 `_模板_` 前缀。

### 3. 列方案（按目录分组）

```markdown
## <目录>/
| 文件 | 判断 | 操作 |
|---|---|---|
| 方案1126.docx | 旧版 | 🗑 删 |
| 方案0313成品.docx | ✅ 最终 | 保留 |
```

末尾汇总 `共 N 文件，保留 X，删 Y，移 Z`。**说"待确认后执行"，不自动跑**。

### 4. 等批准 → 执行

按确认方案做。完成 `tree` 展示最终结构。

## 永远不动

- `_project.yaml` `_meta.yaml` `_index.md` `README.md` `.claude/`
- `data/` 下核心数据（xlsx / shp / 等）
