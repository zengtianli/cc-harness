复盘本轮对话，生成结构化回顾，更新记忆，输出文件。

## 触发时机

每次较长对话结束前、切换任务前、或用户主动 `/recap` 时执行。

## 步骤

### Step 1: 回顾本轮对话

通读整段对话，提取：

- **主题**：一句话概括本轮做了什么
- **目的**：为什么要做这个
- **进程**：按时间顺序列出关键节点（做了什么 → 结果如何）
- **最终成果**：产出了哪些文件/命令/配置/代码
- **踩坑经历**：走了哪些弯路，为什么走弯路，正确做法是什么

### Step 2: 改进与经验

分析：

- **做对了什么**：值得保留的模式、决策
- **做错了什么**：CC 的错误（理解偏差、没查就问、方向错误等）
- **工程模式**：本轮发现/确认的可复用模式
- **沟通层面**：用户反馈了什么，CC 应该怎么调整

### Step 3: CC 侧记录

检查是否需要更新以下内容（如果需要就更新，不需要就跳过）：

1. **Memory**（`~/.claude/projects/*/memory/`）
   - 新的 feedback？用户纠正了 CC 什么行为？
   - 新的 project 信息？项目状态、架构变化？
   - 新的 user 偏好？
   - 新的 reference？外部资源位置？
   - 检查现有记忆是否需要更新或删除

2. **Skills**（`~/Dev/cc-configs/skills/`）
   - 现有 skill 需要更新吗？
   - 需要新建 skill 吗？

   **Skill 候选池**（`~/Dev/cc-configs/skill-candidates.md`）
   
   分析本轮对话，识别是否有**重复出现的工作模式**尚未被 skill/command 覆盖：
   
   - 是否有多步操作被手动执行了 2+ 次？
   - 是否有固定的判断逻辑被反复描述？（比如"先查 X，如果没有就 Y"）
   - 是否有特定领域的知识被反复注入？（比如每次都要解释某个业务规则）
   
   如果发现候选模式：
   1. 读取 `~/Dev/cc-configs/skill-candidates.md`（如不存在则创建）
   2. 检查是否已有类似候选
   3. 如果是新候选，追加一条记录：
      ```
      ## {候选名称}
      - 发现时间: {date}
      - 出现次数: 1
      - 状态: active
      - 模式描述: {什么场景下做什么}
      - 涉及项目: {project}
      - 潜在 skill 类型: command / skill / agent
      ```
   5. 当候选被正式创建为 skill/command 后，更新状态为 `done`；用户明确不需要的标记为 `dismissed`。有状态标记的候选不再触发通知。
   4. 如果已有类似候选，更新出现次数
   5. 如果某候选出现次数 >= 3，提示用户："候选 {name} 已出现 {N} 次，建议正式创建 skill"

   **Skill/Command 使用追踪**（`~/Dev/cc-configs/skill-tracker.json`）

   更新本轮使用过的 skill/command 的追踪数据：
   1. 读取 `~/Dev/cc-configs/skill-tracker.json`
   2. 对本轮使用过的每个 skill/command：
      - 更新 `last_used` 为今天日期
      - `use_count` +1
      - 如果用户纠正了该 skill/command 的行为：`correction_count` +1，更新 `last_correction`，在 `notes` 追加纠正内容摘要
   3. 写回文件

   如果某个 skill/command 的 `correction_count >= 3`，提示用户："skill {name} 已被纠正 {N} 次，建议重写"

3. **Commands**（`~/Dev/cc-configs/commands/`）
   - 现有 command 需要更新吗？
   - 需要新建 command 吗？

4. **CLAUDE.md**（当前项目的）
   - 有新的约束或路径变化需要记录吗？

对每个需要更新的项，直接执行更新。

### Step 4: 用户侧记录

生成 session retro 文件：

**路径**：`~/Dev/stations/docs/knowledge/session-retro-{YYYYMMDD}.md`
- 如果同一天有多个 session，追加后缀：`session-retro-{YYYYMMDD}-2.md`

**格式**（参考已有的 session-retro-20260323.md）：

```markdown
# 会话复盘 · {YYYY-MM-DD}

> 主题：{一句话主题}

---

## 1. 做对了什么

### {做对的事 1}
{具体描述}

**值得保留的模式**：{提炼出的可复用经验}

---

## 2. 走了哪些弯路（和为什么）

### 弯路一：{简述}

{具体过程}

**正确做法**：{应该怎么做}

**根因**：{为什么走了弯路}

---

## 3. 值得学习的工程模式

### {模式名}
{描述}

---

## 4. 沟通层面的反思

{CC 和用户的互动有什么可以改进的}

---

## 5. 成果清单

| 产物 | 路径 | 说明 |
|------|------|------|
| {文件/命令/配置} | {路径} | {一句话说明} |

---

## 6. 未完成项

- [ ] {待办 1}
- [ ] {待办 2}
```

### Step 5: 输出汇总

向用户展示简短汇总：

```
📋 本轮复盘完成

主题：{主题}
成果：{N} 个文件产出
记忆：更新 {N} 条 / 新增 {N} 条
技能候选：{N} 个新增 / {N} 个达到创建阈值
追踪：更新 {N} 个 skill/command 使用记录
待办：{N} 项未完成

文件：~/Dev/stations/docs/knowledge/session-retro-{date}.md
```

## 规则

- 复盘内容要诚实，CC 做错的就说做错了，不要粉饰
- 踩坑经历要写清「根因」，不只是描述现象
- 用户的原话反馈（特别是批评）要如实记录，不要委婉化
- Memory 更新要具体，不要写空泛的"用户喜欢 X"
- 如果本轮没有踩坑，第 2 节可以简写甚至跳过
- 如果本轮没有新的工程模式，第 3 节可以跳过
- retro 文件用中文撰写
