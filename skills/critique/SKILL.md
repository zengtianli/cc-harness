# /critique · 工作流激进审计

**核心理念**：用户不是来求表扬的，是来求进步的。批评是信号，奉承是噪音。

always-on 的指正写在 `feedback_proactive_critique.md`（每轮会话随时触发）；`/critique` 是**按需深度扫**，用户主动问的时候跑一遍完整审计。

---

## 什么时候跑

用户说以下任一：
- `/critique`
- "帮我审一下" / "挑挑刺" / "指正" / "哪里不对"
- "我这样做专业吗 / 对吗"
- "有没有优化空间"
- 会话结束前用户问"这轮做得怎么样"

**不跑**：用户没问、会话进行中、任务还没完成。

---

## 6 维度审计

每维度扫本会话 + 最近 1-2 个会话（走 cclog MCP 搜历史），找到问题就**明确列出**（问题 + 根因 + 修法三段），没问题就说"这维度干净"。

### 1. 决策效率

- 同一个决策反复了几轮？（plan 文件计数 / session 切换次数）
- 有没有模糊词没澄清就开干（"统一 / 合并 / 差不多"）
- 拍板后有没有反悔 → 重做

### 2. 工具复用

- 手工 bash 做了 skill / slash command 能做的事？
- 没用 `/warmup` / `/handoff` / `/health` / `/audit` / `/deploy` / `/ship` 等既定工具
- 重写了已有逻辑（应该 `Grep` 找现成的）

### 3. 协议遵守

- 破坏性操作跳过 plan mode
- commit 没走 audit gate（强推 / `--no-verify`）
- 多步任务没开 TaskCreate 追踪
- 改 SSOT 没跑 build-* / audit 刷派生产物

### 4. 资源浪费

- 远端 session / 子 agent 跑了本地能干的活
- 远端成果没同步回来（如本轮 Ultraplan 废掉）
- 多次 `git fetch` / `curl` 等重复网络请求（应缓存 / 并行）
- 启了 `Agent` 但只做单次查询（应直接 `Read` / `Grep`）

### 5. 归档与接力

- HANDOFF 放错位置（全局根 vs 项目目录）
- 改完没更新 CLAUDE.md / playbook
- 新踩的坑没进 memory（下次还会踩）
- 立了新工具没写 CLAUDE.md 说明节

### 6. 命名与结构

- 文件命名不一致（下划线 vs 连字符 vs 驼峰混用）
- 目录归类漂移（应在 `tools/` 却在根）
- 裸绝对路径未入 `paths.yaml`（高频）
- SSOT 有多份（之前立过的没合并，新的又开一个）

## 输出格式

```
# /critique 审计报告

## 本会话扫到 N 条问题

### 1. [问题名]
- **问题**：具体描述（引用文件路径 / commit / 时间戳）
- **根因**：为什么发生
- **修法**：一句话 + 对应的 skill / 命令

### 2. ...

## 最近会话（cclog 查）

（类似格式）

## 评价

- 本轮干净度：N/6 维度过线
- 最值得改的 1 条：__
- 下次会话入场动作：__
```

---

## 行为守则

- **不软化**：不要 "可能你已经知道" / "如果你不介意" / "仅供参考"
- **不客气**：批的是行为/决策/流程，不是人。直接列事实
- **不放水**：实在没问题就写"这轮干净，X 维度都稳，没可指的"，**不要**硬挑毛病凑数
- **给具体修法**：每条批评必须附可执行的下一步
- **承认 Claude 自己的错**：如果问题出在 Claude（理解歪、工具选错），老实列进报告

---

## 反面（不该做的 critique）

❌ "整体做得不错，有几个小建议..."（奉承开场）
❌ "建议考虑一下是否可以优化..."（缓冲语）
❌ "你 XX 的习惯不好"（批人不批事）
❌ "感觉好像有点..."（模糊）
❌ 列 10 条鸡毛蒜皮凑数

✓ "本会话 3 个实打实的问题：..."
✓ "这条是 Claude 的锅，不怪你"
✓ "本轮干净，6 维度过 5，只有命名一项有 1 条漂移"
